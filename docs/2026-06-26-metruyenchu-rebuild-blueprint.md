## 1. Tổng Quan & Tech Stack

### 1.1 Mô tả

Nền tảng đọc truyện chữ (web novel) tiếng Việt. User đăng ký, đọc truyện, theo dõi, bình luận. Uploader đăng và quản lý truyện. Admin kiểm duyệt và có thể kích hoạt gen audio TTS cho từng chapter qua job queue. Hỗ trợ monetization (coin, VIP), rankings, badges, booklist.

**Cơ hội thị trường:** `metruyencv.com` (trang gốc) đã đóng cửa tháng 02/2026 — lượng user lớn đang tìm nơi mới.

### 1.2 Tech Stack

| Layer | Công nghệ |
|-------|-----------|
| **Frontend** | Next.js 14 (App Router), TypeScript, TailwindCSS, React Query (TanStack Query), Zustand |
| **Audio Library** | howler.js (basic), wavesurfer.js (read-along mode) |
| **PWA** | Service Worker with Workbox, IndexedDB (idb library) |
| **Backend Gateway** | Spring Cloud Gateway (reactive, non-blocking) |
| **Backend Services** | Java 21, Spring Boot 3.4+, Spring Security 6, Spring Data JPA, Flyway |
| **Database** | PostgreSQL 16 (per service) |
| **Cache + Job Queue** | Redis 7 (cache) + Redis Streams (job queue) |
| **TTS Engine** | LocalAI + OmniVoice Q8_0 model (~945MB), GPU 4GB VRAM minimum |
| **Voice Cloning** | OmniVoice voice cloning mode for best Vietnamese quality |
| **Audio Storage** | MinIO (S3-compatible), presigned URLs |
| **Inter-service Comm** | OpenFeign (sync) + RabbitMQ (async events) |
| **Observability** | Micrometer Tracing → OpenTelemetry → Grafana Tempo (traces), Loki (logs), Prometheus (metrics) |
| **Container** | Docker Compose (dev), Kubernetes (prod) |

**Database per Service:**

| Service | Database |
|---------|----------|
| Auth Service | PostgreSQL `auth_db` (users, roles) |
| Story Service | PostgreSQL `story_db` (stories, chapters, categories, tags) |
| Social Service | PostgreSQL `social_db` (comments, bookmarks, follows, ratings, history) |
| Audio Service | PostgreSQL `audio_db` (audio_jobs, audio_files) + Redis + MinIO |
| Notification Service | Redis (temporary) + optional PostgreSQL for history |
| Analytics Service | Reads from all DBs (read replicas) |

### 1.4 Service Communication

```
Sync (OpenFeign + Resilience4j):
  API Gateway → all services (via Feign)
  Story Service → Auth Service (get author info)
  Story Service → Social Service (get ratings count)
  Story Service → Audio Service (get audio status)

Async (RabbitMQ events):
  Story Service publishes: "ChapterPublished" → Notification Service sends push
  Audio Service publishes: "AudioReady" → Notification Service notifies uploader
  Social Service publishes: "NewComment" → Notification Service notifies author
  All services → Analytics Service (ingest events)
```

### 3.3 SWOT Analysis

**Strengths:**
- metruyencv.com (trang gốc) đã đóng cửa — user đang tìm chỗ mới
- Thị trường audio/TTS còn trống, chưa ai làm tốt

- UX sạch, không quảng cáo (giữ điểm mạnh của metruyenchu hiện tại)

**Weaknesses:**
- NovelToon đã có hệ sinh thái hoàn chỉnh
- Phải build từ đầu, không có sẵn content
- TTS cần GPU, chi phí vận hành

**Opportunities:**
- User từ metruyencv.com cũ đổ về
- Audio novel = blue ocean
- Community features (booklist, comment, ranking) chưa ai làm tốt
- Có thể bắt đầu với focus vào convert/dịch sạch

**Threats:**
- Pháp lý: site có thể bị chặn
- Copyright issues với truyện dịch không phép
- NovelToon đang thống trị

### 4.8 Audio TTS System

Core killer feature. Chi tiết dựa trên research LocalAI + OmniVoice.

**Model config:**
- LocalAI image: `localai/localai:latest-gpu-nvidia-cuda-12`
- OmniVoice model: `omnivoice-cpp` (Q8_0, ~945MB) hoặc HQ (BF16, ~1.6GB)
- GPU tối thiểu: 4GB VRAM (GTX 1650+), khuyến nghị: RTX 3060+ (8-12GB)
- Voice cloning mode recommended cho tiếng Việt (cần reference audio + transcript)
- Fine-tune models available: `splendor1811/omnivoice-vietnamese`
- Alternative: FPT.AI TTS (trả phí, chất lượng cao nhất cho tiếng Việt)
- RTF: ~0.025 trên GPU (40x real-time), ~1-2 giây cho 1 phút audio

#### 4.8.1 Flow (Split-Merge Architecture)

```
Admin → Chapter Detail → Chọn Voice Profile từ dropdown → Click "Generate Audio"
  │
  ▼
POST /api/v1/audio/jobs { chapterId, voiceProfileId }
  │
  ▼
Audio Service:
  1. Fetch chapter content (Feign → Story Service)
  2. Split content → N segments (500-2000 ký tự/segment, theo paragraph)
  3. Create parent_job (status=PROCESSING)
  4. Create N segment_jobs (status=PENDING)
  5. Push N messages to RabbitMQ queue "audio.segments"
  │
  ▼
TTS Segment Workers (N consumers, parallel, horizontal scaling):
  [Worker 1] ← seg_0 ──┐
  [Worker 2] ← seg_1 ──┤
  [Worker 3] ← seg_2 ──┤── Song song, mỗi worker xử lý 1 segment
  [Worker 1] ← seg_3 ──┤
  [Worker 2] ← seg_4 ──┘
  │ Mỗi worker:
  │   • UPDATE segment_job → PROCESSING
  │   • Gọi LocalAI /v1/audio/speech (với voice params từ profile)
  │   • Upload MP3 → MinIO: segments/{chapterId}/v{ver}/seg_{idx}.mp3
  │   • UPDATE segment_job → COMPLETED
  │   • Publish "segment.completed" event
  │
  ▼
Khi count(COMPLETED) == total_segments → Trigger Merge
  │
  ▼
Merge Worker (RabbitMQ consumer "audio.merge"):
  • Download all segments từ MinIO
  • FFmpeg concat segments → merged.mp3
  • Upload merged → MinIO: chapters/{chapterId}/v{ver}.mp3
  • CREATE AudioFile record
  • UPDATE parent_job → COMPLETED, chapter.has_audio = true
  • Publish "audio.job.completed" → Notification Service
  │
  ▼
Frontend polls parent_job progress → COMPLETED → show audio player
```

## 10. Architecture & Technical Decisions

### 10.1 Spring Microservices Patterns

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **API Gateway** | Spring Cloud Gateway | Non-blocking, reactive, better throughput than Zuul |
| **Service Discovery** | K8s native (if on K8s) / Eureka (if VMs) | K8s built-in, no extra service |
| **Inter-service sync** | OpenFeign + Resilience4j | Circuit breaker + retry + timeout |
| **Inter-service async** | RabbitMQ | Lightweight, easy ops. Kafka if >100K msg/s |
| **Auth flow** | JWT passthrough | Gateway verifies JWT, forwards userId/roles in headers |
| **Internal JWT** | Skip re-verification (trust gateway) | Services trust `X-User-Id`, `X-User-Roles` headers |
| **Distributed transaction** | Saga + Outbox pattern | Eventual consistency, no 2PC |
| **Project structure** | Gradle multi-module mono-repo | Team < 20, shared libs, atomic commits |
| **Database per service** | Yes | Separate PostgreSQL per service, no shared DB |
| **Migration** | Flyway | Spring-native, SQL files, git-friendly |
| **API docs** | SpringDoc OpenAPI per service + Gateway aggregator | Single Swagger UI |

### 10.2 TTS / LocalAI Decisions

| Decision | Choice |
|----------|--------|
| **TTS Model** | LocalAI + OmniVoice Q8_0 (~945MB) |

## 11. Security & Compliance

| Category | Measure | Implementation |
|----------|---------|----------------|
| **Authentication** | JWT (RS256) | Access token 15 min, Refresh token 7 days (httpOnly cookie) |
| **OAuth2** | Social login | Google, Facebook, Zalo — Spring Security OAuth2 Client |
| **Password** | BCrypt | BCryptPasswordEncoder, strength=10 |
| **2FA** | TOTP | Optional, Spring Security + Google Authenticator |
| **Authorization** | Role + Permission | `@PreAuthorize("hasRole('ADMIN')")` + method-level security |
| **Input validation** | Bean Validation | `@NotBlank`, `@Email`, `@Size`, custom validators |
| **Rate limiting** | Bucket4j | Login: 5/min, API: 100/min per IP, per user tier |
| **CORS** | Whitelist | Only allow frontend domain |
| **CSRF** | Disabled | JWT-based, no cookie sessions |
| **SQL Injection** | JPA parameterized | Spring Data JPA prevents injection |
| **XSS** | React + CSP | React auto-escapes, Content-Security-Policy headers |
| **File upload** | Validation | Images only (JPEG/PNG/WebP), max 5MB, virus scan |
| **Audio files** | Signed URLs | MinIO presigned URLs, expire 1 hour |
| **HTTPS** | Enforced | Redirect HTTP → HTTPS, HSTS headers |
| **Audit logging** | Admin actions | All admin CRUD logged to activity_logs |
| **GDPR** | Data export + delete | Export user data, soft delete with 30-day recovery |

## 12. Performance & Caching

| Layer | Strategy | Details |
|-------|----------|---------|
| **Database** | Indexing | Composite indexes for listing queries, partial indexes for soft delete |
| **Database** | Connection pool | HikariCP (max 20 connections per instance) |
| **Database** | Pagination | Keyset pagination for large lists (chapters), offset for small |
| **Cache** | Redis | Hot stories (5min TTL), category list (1h TTL), user sessions, config |
| **Cache** | Cache-aside | Read-through cache, invalidate on write |
| **Frontend** | ISR + SSR | Homepage ISR 300s, story detail SSR, reader Streaming SSR |
| **Frontend** | Image optimization | next/image, WebP/AVIF format, lazy loading |
| **Frontend** | Code splitting | Dynamic imports for AudioPlayer, Admin dashboard, Rich editor |
| **Frontend** | Route prefetch | Next.js automatic `<Link>` prefetch |
| **Audio** | CDN | Cloudflare proxy for audio file delivery |
| **Audio** | Range requests | HTTP Range header for seekable streaming |
| **Audio** | Preloading | Preload next chapter audio when current 75% done |
| **API** | Response compression | Gzip/Brotli compression on nginx/gateway level |
| **API** | ETag + If-None-Match | Conditional GET for cache validation |

### 13.2 Kubernetes (Production)

- **Service mesh:** Optional (Istio / Linkerd if needed)
- **Config:** Helm charts + ConfigMap + Vault (secrets)
- **CI/CD:** GitHub Actions → Docker registry → Helm deploy
- **Monitoring:** Prometheus + Grafana (metrics), Loki (logs), Tempo (traces)
- **Backup:** pg_dump cron for each DB, MinIO backup to S3

### 13.3 CI/CD Pipeline

```
1. Push to GitHub → GitHub Actions
2. Gradle build (compile, test, lint, contract tests)
3. npm build (typecheck, lint, test)
4. Docker build for each service
5. Push to Docker registry (GHCR / Docker Hub)
6. Helm upgrade to staging
7. Smoke tests (cypress / karate)
8. Helm upgrade to production (manual approval)
```
